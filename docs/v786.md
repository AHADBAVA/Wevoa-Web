# WevoaWeb v786

This document is the versioned platform snapshot for **WevoaWeb Runtime 786.0.0**.

It is meant to answer one simple question clearly:

**What can WevoaWeb do right now as a language, framework, runtime, CLI, and installable developer platform?**

It is not a future roadmap. It is a current-state A-to-Z reference for the platform as it exists in this repository.

## 2. What WevoaWeb Is

WevoaWeb is an original **server-first programming language and web platform** built in modern C++17.

It combines:

- a custom scripting language
- a lexer, parser, AST, and interpreter
- a server-side template engine
- an HTTP server
- a developer CLI
- a build and serve workflow
- a package system
- a Windows installer path

In practical terms, it is not just “a language” and not just “a framework.” It is a compact vertical stack where one runtime is responsible for project creation, route execution, template rendering, request handling, packaging, and developer tooling.

---

## 3. Core Philosophy

WevoaWeb is built around a few strong ideas:

- HTML-first application delivery
- server-side rendering by default
- minimal browser-side complexity
- no JavaScript dependency for normal app flow
- one binary ecosystem
- readable, hackable architecture

This gives the platform a very different feel from modern browser-heavy stacks. The browser gets HTML and assets. The runtime does the work on the server side.

## 4. Language Capabilities

### 4.1 Core Syntax

The language supports:

- `let`
- `const`
- `func`
- `route`
- `component`
- `if / else`
- `loop`
- `while`
- `break`
- `continue`
- `return`
- `import`
- `html { ... }`

### 4.2 Runtime Value Types

Current runtime values include:

- `nil`
- integer
- string
- boolean
- array
- object
- callable

Internally, values are represented through a `std::variant`-based runtime model.

### 4.3 Operators

Supported operators include:

- arithmetic: `+`, `-`, `*`, `/`
- comparison: `>`, `<`, `>=`, `<=`, `==`, `!=`
- logical: `&&`, `||`, `!`
- assignment: `=`
- property access: `.`
- indexing: `[]`
- function calls: `()`

### 4.4 Arrays and Objects

The language supports:

- array literals
- object literals
- nested array/object values
- array indexing
- object property access
- object key access through brackets

Examples:

```text
let users = ["Ahad", "Ali"]
let user = { name: "Ahad", role: "admin" }
let first = users[0]
let name = user.name
```

### 4.5 Scope and Closures

WevoaWeb uses lexical scoping with chained environments.

That means:

- blocks create child scopes
- functions capture the environment where they are declared
- routes also capture runtime scope
- constants are enforced per binding

Closures already work in the current interpreter model.

### 4.6 Imports

The import system supports:

- relative file imports
- package imports with `@package`

Examples:

```text
import "shared.wev"
import "@auth"
```

Imported source is parsed and executed inside the runtime session, with duplicate imports skipped safely.

## 5. Template Engine Capabilities

WevoaWeb has a real server-side template engine.

### 5.1 Interpolation

Templates support:

- `{{ variable }}`
- `{{ object.field }}`
- `{{ array[0] }}`
- expressions inside interpolation

### 5.2 Template Control Flow

Supported directives include:

- `{% if condition %}`
- `{% else %}`
- `{% for item in items %}`
- `{% end %}`

### 5.3 Layout System

Templates support:

- `extend "layout.wev"`
- `section content { ... }`
- section injection into layouts

This gives the platform a Blade-like layout workflow while keeping the implementation original.

### 5.4 Includes

Reusable partial templates are supported with:

- `include "partials/nav.wev"`

### 5.5 Components

The framework supports reusable template components declared in backend source and rendered in templates.

Example shape:

```text
component card {
html {
<article>{{ title }}</article>
}
}
```

### 5.6 Compiled Template Cache

Templates are no longer interpreted in the slowest possible way every request.

Current runtime behavior includes:

- compiled template node trees
- template cache
- fragment cache
- build-time template manifests
- dev invalidation on reload
- production reuse of compiled templates

## 6. Web Framework Capabilities

### 6.1 Routing

Routes are declared directly in the language.

Supported method handling includes:

- `GET`
- `POST`
- `PUT`
- `PATCH`
- `DELETE`

### 6.2 Dynamic Parameters

Route patterns support:

- `:id`
- multiple dynamic segments

Example:

```text
route "/tasks/:id/update" method POST {
  return params.id
}
```

### 6.3 Middleware

Routes support middleware binding:

```text
route "/dashboard" middleware auth_require {
  return view("dashboard.wev")
}
```

Middleware can:

- allow execution
- short-circuit with a response
- redirect unauthenticated users

### 6.4 Request Object

The request object already exposes:

- `request.method`
- `request.method()`
- `request.path`
- `request.target`
- `request.body`
- `request.query("key")`
- `request.form("key")`
- `request.json()`
- `request.csrf()`
- uploaded file access

### 6.5 Response Helpers

The runtime includes response helpers:

- `json(...)`
- `redirect(...)`
- `status(code, body[, contentType])`

This gives routes structured response behavior instead of only raw HTML strings.

## 8. Database Capabilities

### 8.1 SQLite

WevoaWeb includes native SQLite support inside the runtime.

Current entry point:

- `sqlite.open(path)`

### 8.2 Query Surface

The SQLite bridge supports:

- `exec`
- `query`
- `scalar`

### 8.3 Parameter Binding

Supported parameter styles:

- positional arrays
- named object bindings

### 8.4 Migrations

The CLI includes:

- `wevoa migrate`
- `wevoa make:migration <name>`

Migration files are SQL-first, simple, and easy to audit.

### 8.5 Concurrency Strategy

The runtime already improved SQLite handling for server usage:

- per-worker-thread connection strategy
- WAL mode support
- safer concurrent access than a naïve shared connection model

SQLite is still the only built-in database backend today, but it is integrated deeply enough to build real apps.

## 15. Current Issues and Limits

WevoaWeb is much stronger now, but it still has real limitations.

### 15.1 Runtime / Performance Limits

- the interpreter is still tree-walk based
- there is no bytecode VM yet
- there is no JIT
- heavy workloads will still hit interpreter overhead

### 15.2 Server Limits

- the platform is stronger under load now, but it is not yet a full async reactor-style runtime
- no HTTP/2 or advanced protocol surface
- not designed yet as a giant high-throughput internet edge server

### 15.3 Database Limits

- SQLite is the only built-in backend
- no PostgreSQL/MySQL driver layer yet
- no ORM
- no query builder

### 15.4 Ecosystem Limits

- package ecosystem is real but still early
- no remote package registry with mature hosting workflow yet
- no large third-party community package set yet

### 15.5 Language Limits

- no floating-point runtime type
- no classes
- no exception model for user code
- language is still intentionally compact

These limits do not make the platform non-functional. They define where the next major growth areas are.

## 16. Best Use Cases Right Now

WevoaWeb is already a strong fit for:

- dashboards
- admin panels
- internal tools
- CRUD systems
- simple account-based web apps
- package-backed starter applications
- educational runtime and language exploration

It is less ideal right now for:

- massive package-registry-dependent ecosystems
- extremely high-throughput async workloads
- applications requiring many database backends out of the box

## 17. Bottom-Line Status

**WevoaWeb v786 is a real working platform.**

At this version, it already provides:

- a custom language
- a full request lifecycle
- templates, layouts, includes, and components
- routing, middleware, sessions, CSRF, and auth helpers
- SQLite persistence and migrations
- package discovery and installation
- clean starter packs
- development and production workflows
- installer-oriented distribution

The platform is no longer just “interesting.” It is now **usable**.

Its biggest strength is integration and clarity.

Its biggest remaining gap is maturity at scale: VM performance, database breadth, and ecosystem depth still need time.

That said, version `786.0.0` is a meaningful release point because it represents a coherent language + framework + runtime + CLI + packaging story in one system.
